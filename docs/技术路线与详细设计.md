# 气象数据管理与分析系统：技术路线与详细设计

本文档面向论文详细设计、答辩说明和后续维护，描述当前代码实现中的技术栈、模块划分、数据流和关键设计决策。

## 1. 总体架构

系统采用前后端分离架构：

```text
Vue 3 前端
  ↓ Axios + JWT
Spring Boot REST API
  ↓
Service 业务层
  ├─ PostgreSQL：业务数据主存储
  ├─ Redis：验证码、站点列表、省份列表等缓存
  └─ HDFS：导出文件、GIS 快照和表数据归档
```

### 1.1 技术选型

| 层次 | 技术 |
|---|---|
| 前端框架 | Vue 3 Composition API、TypeScript |
| 前端 UI | Element Plus、Element Plus Icons |
| 路由与状态 | Vue Router、Pinia |
| 可视化 | ECharts、百度地图 GL、MapVGL |
| HTTP | Axios，统一拦截 JWT 和错误 |
| 后端框架 | Spring Boot 3.5.7 |
| 安全 | Spring Security、JWT、`@PreAuthorize` |
| 持久化 | MyBatis、PostgreSQL |
| 缓存 | Spring Data Redis、Lettuce |
| 文件存储 | Hadoop HDFS Client 3.3.6 |
| 报表 | Apache POI，生成 DOCX/XLSX |
| 预测 | Apache Commons Math |
| 邮箱 | Spring Mail，QQ SMTP |

### 1.2 路由结构

```text
/welcome
/login
/register
/dashboard
  /profile
  /data-collection
  /data-analysis/statistical
  /data-analysis/gis-map
  /data-prediction
  /personalized-service
/admin
  /user-management
  /storage-management
```

前端路由守卫基于 `localStorage` 中的 `token` 和 `role` 判断访问权限：

- `role=0`：普通用户，访问 `/dashboard/**`。
- `role=1`：管理员，访问 `/admin/**`。
- 已登录用户访问登录/注册页时，会按角色跳转到对应首页。

## 2. 用户认证与权限

### 2.1 注册流程

1. 用户在注册页填写邮箱并请求验证码。
2. `AuthController.send-email-code` 调用邮箱服务生成验证码。
3. 验证码和发送锁写入 Redis。
4. 用户提交注册表单。
5. 后端校验验证码、用户名、密码、邮箱等字段。
6. 密码通过 BCrypt 加密后写入 `sys_user`。
7. 注册成功返回 JWT 和用户信息。

公开注册强制创建普通用户，管理员由后台或数据库分配。

### 2.2 登录与 JWT

登录接口根据用户名查询用户，使用 BCrypt 校验密码，通过后生成 JWT。JWT 中携带：

- `sub`：用户名
- `userId`：用户 ID
- `role`：角色

`JwtAuthenticationFilter` 在 Spring Security 过滤器链中解析 `Authorization: Bearer <token>`，注入 `SecurityContext`。

### 2.3 权限控制

| API | 权限 |
|---|---|
| `/api/auth/login`、`/api/auth/register`、`/api/auth/send-email-code` | 匿名可访问 |
| `/uploads/**` | 匿名可访问 |
| `/api/admin/**` | 管理员 |
| `/api/storage/**` | 管理员 |
| 其他 `/api/**` | 登录用户 |

## 3. 数据采集模块

普通用户通过数据采集页创建采集任务。核心流程：

```text
DataCollection.vue
  → POST /api/collection-tasks
  → CollectionTaskService.createTask()
  → 写入 collection_tasks
  → WeatherApiService 调用 CMA 接口
  → 解析观测数据
  → 批量写入 meteor_data
  → 更新任务状态
```

`collection_tasks` 使用 PostgreSQL 数组保存站点和要素列表：

- `station_list INTEGER[]`
- `element_list VARCHAR(20)[]`

后端通过自定义 TypeHandler 完成 Java 数组和 PostgreSQL 数组的转换。

## 4. 统计分析模块

统计分析页 `StatisticalAnalysis.vue` 使用以下 API：

| 功能 | API | 数据源 |
|---|---|---|
| 概览统计 | `GET /api/meteor-data/summary` | `meteor_data`、`collection_tasks` |
| 时间序列 | `GET /api/meteor-data/time-series` | `meteor_data` |
| 空间分布 | `GET /api/meteor-data/spatial-map` | `meteor_data` + `weather_stations` |
| 相关性 | `POST /api/meteor-data/correlation` | `meteor_data` |
| 明细表格 | `GET /api/meteor-data/records` | `meteor_data` |
| Word 报告 | `POST /api/meteor-data/export/analysis-docx` | 生成并归档到 HDFS |
| Excel 明细 | `POST /api/meteor-data/export/records-xlsx` | 生成并归档到 HDFS |

### 4.1 时间序列

后端按 `hour/day/week/month` 粒度聚合，返回每个时间桶的平均值、最小值和最大值。

### 4.2 相关性热力图

后端按 `datetime + stationId` 对齐多要素数据，计算皮尔逊相关系数矩阵，前端使用 ECharts heatmap 展示。前端已调整图表底部间距，避免 x 轴标签与视觉组件重合。

### 4.3 报表导出与 HDFS 归档

普通用户导出报表时，浏览器仍会下载文件，同时后端把同一份文件写入 HDFS：

```text
/weather-platform/exports/reports/user-{用户ID}/analysis/weather-analysis-report-{时间戳}.docx
/weather-platform/exports/reports/user-{用户ID}/records/weather-data-records-{时间戳}.xlsx
```

实现类：`ReportExportService`。

## 5. GIS 可视化模块

GIS 页面 `WeatherMapView.vue` 基于百度地图 GL 和 MapVGL 实现：

- 站点 Marker 展示。
- 温度、相对湿度、3 小时降水量等要素可视化。
- 观测时间选择。
- 地图点选查询最近站点。
- 最近站点、气象数据和位置详情弹窗展示。

主要数据源：

- `weather_stations`
- `history_weather_data`

主要 API：

- `GET /api/weather/stations/all`
- `GET /api/weather/element-data`
- `GET /api/weather/observation-times`
- `GET /api/weather/latest-observation-time`
- `GET /api/weather/station/{stationId}/data`
- `GET /api/weather/grid-data`
- `GET /api/weather/grid-data/point`

## 6. 数据预测模块

预测模块读取用户采集数据 `meteor_data`，支持：

- 线性回归
- 移动平均
- 指数平滑

主要 API：

- `GET /api/prediction/algorithms`
- `GET /api/prediction/data-info`
- `POST /api/prediction/run`

返回内容包括历史序列、预测序列、MAE/RMSE/R² 等指标和阈值预警。

## 7. 个性化服务模块

个性化页面综合展示用户数据概览、最新数据、要素分布和界面偏好。

偏好主要保存在浏览器 localStorage：

- 时间粒度

- 预测步数

主题配置由 Pinia `theme` store 维护，并通过 CSS 变量同步到 Element Plus 与自定义样式。

## 9. 管理员存储管理

存储管理页分为两个 Tab：

1. HDFS 工作区
2. Redis 缓存

### 9.1 HDFS 工作区

页面管理范围固定在 `hdfs.base-path=/weather-platform`，避免误操作整个 HDFS 根目录。

支持能力：

- HDFS 连接状态、NameNode URI、SafeMode、平台目录状态。
- 总容量、已用容量、可用容量、使用率。
- 文件浏览、面包屑路径、返回上级。
- 新建目录、上传文件、下载文件、重命名、删除。
- 初始化平台目录。
- GIS 展示数据归档。
- GIS 数据库表全量上传。
- 文件元数据展示：副本数、块大小、估算块数、HDFS 占用、权限、Owner、Group、修改时间。

目录没有副本数和分块概念，页面中显示为 `-`；目录大小使用 HDFS `ContentSummary` 递归统计。

### 9.2 HDFS 目录约定

| 目录 | 用途 |
|---|---|
| `/raw-data` | 原始数据 |
| `/raw-data/meteor` | 气象采集原始数据预留 |
| `/raw-data/history` | 历史数据预留 |
| `/raw-data/gis` | GIS 快照 |
| `/raw-data/gis/tables` | GIS 依赖表全量导出 |
| `/exports` | 报表和导出结果 |
| `/backups` | 备份 |
| `/temp` | 临时文件 |

### 9.3 Redis 缓存

Redis 页面展示：

- Redis 版本
- 当前连接数
- 运行时长
- 匹配 Key 数
- Key 类型
- TTL

支持按模式查询和清理，例如 `weather:*`。

## 10. HDFS 数据归档设计

### 10.1 GIS 页面快照

管理员点击“归档 GIS 数据”后，后端读取 GIS 页面所需站点和要素数据，生成 JSON：

```text
/weather-platform/raw-data/gis/weather-gis-{element}-{time}.json
```

### 10.2 GIS 表数据全量上传

管理员点击“上传 GIS 表数据”后，后端导出：

- `weather_stations.json`
- `history_weather_data.json`
- `manifest.json`

目标路径：

```text
/weather-platform/raw-data/gis/tables/export-{yyyyMMddHHmmss}/
```

说明：当前不再导出已删除的 `weather_data`。

### 10.3 普通用户报表归档

普通用户在统计分析页导出的 Word/Excel 会同步写入 `/exports/reports/user-{id}`，便于管理员在 HDFS 管理页审计和下载。

## 11. Redis 使用

Redis 主要用于：

- 邮箱验证码，5 分钟过期。
- 邮箱验证码发送锁，约 60 秒防重复发送。
- 气象站全量列表缓存。
- 省份列表缓存。

管理员 Redis 页仅做状态查看和 Key 管理，不直接参与 HDFS 文件操作。

## 12. 数据库设计变化

当前实现保留 5 张核心表，已删除旧的 `weather_data` 表及相关代码。GIS 页面和 GIS 表导出使用：

- `weather_stations`
- `history_weather_data`

统计分析、预测和普通用户报表使用：

- `collection_tasks`
- `meteor_data`

## 14. 当前限制和后续建议

1. HDFS 文件块数当前按文件大小和 block size 估算，未读取 LocatedBlocks 明细。
2. `meteor_data.station_id` 与 `weather_stations.station_id` 类型不同，后续可统一。
3. 普通用户报表归档失败会影响导出接口返回，若需要更高可用性，可改为异步归档或失败降级。
4. `application.properties` 当前包含开发环境连接信息，生产环境应改为环境变量或本地配置覆盖。
