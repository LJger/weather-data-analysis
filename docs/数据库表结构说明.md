# 数据库表结构说明

本文档基于 `back_end/src/main/resources/sql/create_tables.sql` 和当前后端实体/Mapper 整理。

## 1. 数据库概览

- 数据库：PostgreSQL
- Schema：`gra_project`
- 当前核心业务表：5 张
- 已移除表：`weather_data`

当前表清单：

1. `sys_user`：系统用户。
2. `weather_stations`：气象站基础信息。
3. `collection_tasks`：气象数据采集任务。
4. `meteor_data`：采集任务产出的用户气象要素数据。
5. `history_weather_data`：GIS 和历史气象查询使用的站点历史观测数据。

## 2. 表结构

### 2.1 `sys_user`

用途：保存用户账号、角色、联系方式和个人资料。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `user_id` | `BIGSERIAL` | PK | 用户 ID |
| `username` | `VARCHAR(30)` | NOT NULL, UNIQUE | 用户名 |
| `password_hash` | `VARCHAR(100)` | NOT NULL | BCrypt 密码哈希 |
| `role` | `SMALLINT` | NOT NULL, DEFAULT `0` | 0=普通用户，1=管理员 |
| `email` | `VARCHAR(100)` | UNIQUE | 邮箱 |
| `phone` | `VARCHAR(20)` | UNIQUE | 手机号 |
| `gender` | `SMALLINT` | DEFAULT `0` | 0=未知，1=男，2=女 |
| `age` | `SMALLINT` |  | 年龄 |
| `region` | `VARCHAR(100)` |  | 地区 |
| `organization` | `VARCHAR(200)` |  | 机构/单位 |
| `bio` | `TEXT` |  | 个人简介 |
| `avatar_url` | `VARCHAR(500)` |  | 头像 URL |
| `created_at` | `TIMESTAMPTZ` | DEFAULT `NOW()` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | DEFAULT `NOW()` | 更新时间 |

关联关系：

- `collection_tasks.user_id -> sys_user.user_id`
- `meteor_data.user_id -> sys_user.user_id`

删除用户时，用户采集任务和采集数据通过外键级联删除。

### 2.2 `weather_stations`

用途：保存气象站基础元数据和地理坐标。GIS 页面、站点筛选和历史气象查询都会使用该表。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `station_id` | `VARCHAR(20)` | PK | 气象站编号 |
| `province` | `VARCHAR(50)` | NOT NULL | 省份 |
| `station_name` | `VARCHAR(100)` | NOT NULL | 站点名称 |
| `latitude` | `INTEGER` | NOT NULL | 纬度乘以 100 |
| `longitude` | `INTEGER` | NOT NULL | 经度乘以 100 |
| `pressure_sensor_altitude` | `NUMERIC(8,2)` |  | 气压传感器海拔 |
| `observation_altitude` | `NUMERIC(8,2)` |  | 观测场海拔 |

关联关系：

- `history_weather_data.station_id -> weather_stations.station_id`
- `meteor_data.station_id` 是整数形式站点号，分析查询中通过类型转换与 `weather_stations.station_id` 关联。

### 2.3 `collection_tasks`

用途：保存普通用户创建的数据采集任务及执行状态。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `task_id` | `BIGSERIAL` | PK | 任务 ID |
| `task_name` | `VARCHAR(255)` | NOT NULL | 任务名称 |
| `user_id` | `INTEGER` | NOT NULL, FK | 创建用户 |
| `start_time` | `TIMESTAMPTZ` | NOT NULL | 采集起始时间 |
| `end_time` | `TIMESTAMPTZ` | NOT NULL | 采集结束时间 |
| `station_list` | `INTEGER[]` | NOT NULL | 采集站点 ID 数组 |
| `element_list` | `VARCHAR(20)[]` | NOT NULL | 气象要素代码数组 |
| `task_description` | `TEXT` |  | 任务描述 |
| `status` | `INTEGER` | NOT NULL, DEFAULT `0` | 0=待执行，1=运行中，2=成功，3=失败 |
| `created_at` | `TIMESTAMPTZ` | DEFAULT `NOW()` | 创建时间 |
| `updated_at` | `TIMESTAMPTZ` | DEFAULT `NOW()` | 更新时间 |

索引：

- `idx_collection_tasks_user_id(user_id)`

实现说明：

- PostgreSQL 数组字段通过自定义 TypeHandler 与 Java 数组互转。
- 任务创建后由 `CollectionTaskService` 调用 CMA 接口并写入 `meteor_data`。

### 2.4 `meteor_data`

用途：保存用户采集任务产生的明细气象要素数据，是统计分析、预测、报表导出的主要数据源。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `id` | `BIGSERIAL` | PK | 记录 ID |
| `user_id` | `INTEGER` | NOT NULL, FK | 用户 ID |
| `task_id` | `BIGINT` | NOT NULL, FK | 来源采集任务 |
| `station_id` | `INTEGER` | NOT NULL | 站点 ID |
| `element_code` | `VARCHAR(20)` | NOT NULL | 要素代码，如 `TEM`、`RHU`、`PRE_3h` |
| `datetime` | `TIMESTAMPTZ` | NOT NULL | 观测时刻 |
| `value` | `NUMERIC(8,2)` | NOT NULL | 观测值 |

索引：

- `idx_meteor_data_user_id(user_id)`
- `idx_meteor_data_task_id(task_id)`
- `idx_meteor_data_datetime(datetime)`
- `idx_meteor_data_user_elem_dt(user_id, element_code, datetime)`

使用场景：

- `/api/meteor-data/summary`
- `/api/meteor-data/time-series`
- `/api/meteor-data/correlation`
- `/api/meteor-data/spatial-map`
- `/api/meteor-data/records`
- `/api/prediction/**`
- 普通用户 Word/Excel 报表导出

### 2.5 `history_weather_data`

用途：保存简化版历史气象观测数据，主要服务 GIS 页面和管理员 GIS 表数据全量上传。

| 字段 | 类型 | 约束/默认值 | 说明 |
|---|---|---|---|
| `station_id` | `VARCHAR(20)` | PK, FK | 气象站编号 |
| `observation_time` | `TIMESTAMP` | PK | 观测时刻 |
| `temperature` | `NUMERIC(6,2)` |  | 气温 |
| `relative_humidity` | `NUMERIC(5,2)` |  | 相对湿度 |
| `precipitation_3h` | `NUMERIC(8,4)` |  | 3 小时降水量 |

主键：

- `(station_id, observation_time)`

索引：

- `idx_history_weather_obs_time(observation_time)`

使用场景：

- GIS 气象要素展示：温度、相对湿度、3 小时降水量。
- 点选地图位置时的最近站点气象信息查询。
- 管理员“归档 GIS 数据”和“上传 GIS 表数据”。

## 3. 表关系

```text
sys_user 1 ── N collection_tasks
sys_user 1 ── N meteor_data
collection_tasks 1 ── N meteor_data
weather_stations 1 ── N history_weather_data
```

`meteor_data.station_id` 当前不直接建立外键到 `weather_stations`，原因是采集任务使用整数站点 ID，而站点基础表使用字符型站号。代码层通过 SQL 类型转换完成关联。

## 4. 已移除的 `weather_data`

旧版本中存在 `weather_data` 多要素观测表，但当前页面和服务没有直接使用该表。根据现有实现已删除：

- `WeatherData` 实体
- `WeatherDataMapper` 接口
- `WeatherDataMapper.xml`
- 建表脚本中的 `weather_data`
- 管理员 HDFS 全量上传中的 `weather_data` 导出

当前 GIS 相关全量上传只导出：

- `gra_project.weather_stations`
- `gra_project.history_weather_data`

## 5. HDFS 中的数据副本

数据库仍是业务查询主存储；HDFS 用于文件化归档和导出结果保存：

| HDFS 路径 | 来源 |
|---|---|
| `/raw-data/gis/weather-gis-*.json` | GIS 当前展示快照 |
| `/raw-data/gis/tables/export-*/weather_stations.json` | `weather_stations` 全量导出 |
| `/raw-data/gis/tables/export-*/history_weather_data.json` | `history_weather_data` 全量导出 |
| `/raw-data/gis/tables/export-*/manifest.json` | 本次表导出清单 |
| `/exports/reports/user-{id}/analysis/*.docx` | 普通用户 Word 分析报告 |
| `/exports/reports/user-{id}/records/*.xlsx` | 普通用户 Excel 明细导出 |

## 6. 维护建议

1. 新增查询优先复用已有索引，尤其是 `meteor_data(user_id, element_code, datetime)`。
2. 若后续大量使用站点关联分析，可评估统一 `meteor_data.station_id` 与 `weather_stations.station_id` 的类型。
3. 历史观测量继续增长时，可评估按时间对 `meteor_data` 和 `history_weather_data` 做分区。
4. HDFS 中的导出文件属于派生数据，数据库表仍是权威数据源。
